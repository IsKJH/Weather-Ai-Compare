01

코드 패턴 비교 — 기능 제외, 코드 규칙만

동일한 스펙을 주었을 때 세 AI가 어떤 코드 패턴을 선택했는지 비교합니다. 어떤 기능을 넣었느냐가 아닌, 그 기능을 어떻게 구현했느냐입니다.

1-1 ViewModel 상태 관리 방식

Kotlin Compose에서 ViewModel 상태를 관리하는 방식은 크게 두 가지입니다. 세 AI가 각각 다른 선택을 했고 안전성 차이가 발생했습니다.

Claude ⚠ 주의
StateFlow + .value = .copy()
MutableStateFlow를 사용하되, 업데이트를 value = value.copy()로 처리.

⚠ 비동기 race condition 위험. 두 코루틴이 동시에 읽고 쓰면 먼저 읽은 값이 덮어씌워질 수 있음.
Codex ✅ 무난
mutableStateOf (Compose-native)
var uiState by mutableStateOf(...)로 Compose 런타임이 직접 상태 변화를 추적. 코드가 가장 간결.

단, ViewModel이 Compose 런타임에 의존하므로 순수 Kotlin 단위 테스트 시 별도 처리가 필요.
Gemini ✅✅ 권장
StateFlow + .update {}
_uiState.update { it.copy(...) }로 원자적 업데이트.

✅ CAS 방식으로 동시성 문제 방지. 코루틴 환경에서 가장 안전한 패턴. 프로덕션에서 권장.
❌ Claude — 비원자적 업데이트 (race condition 위험)
fun selectCity(index: Int) {
    _uiState.value = _uiState.value.copy(  // 읽기
        selectedIndex = index,             // 쓰기 사이에 gap 존재
        weatherData = mockWeatherList[index]
    )
}
✅ Gemini — 원자적 update {} (안전)
fun selectCity(city: String) {
    val weather = mockWeatherList
        .find { it.city == city } ?: return
    _uiState.update {         // 원자적 람다
        it.copy(weatherData = weather)
    }
}
1-2 Composable 파라미터 설계

Composable에 ViewModel을 직접 넘기면 @Preview 사용이 불가능하고 단위 테스트가 어려워집니다. 상태(UiState)와 이벤트(콜백)만 파라미터로 받는 순수 함수 형태가 올바른 설계입니다.

❌ Claude / Gemini — ViewModel 직접 주입
// @Preview 불가, 단위 테스트 어려움
@Composable
fun WeatherScreen(
    viewModel: WeatherViewModel
) {
    val uiState by viewModel.uiState
        .collectAsStateWithLifecycle()
    // viewModel 메서드를 직접 호출
}
✅ Codex — 순수 Composable (UiState + callbacks)
// @Preview 가능, 테스트 용이
@Composable
fun WeatherScreen(
    uiState: WeatherUiState,
    onCitySelected: (Int) -> Unit,
    onRefresh: () -> Unit,
    onFavoriteToggle: () -> Unit
) { ... }
1-3 내부 Composable의 private 수식자

파일 내부에서만 쓰는 Composable은 private으로 선언해야 합니다. 외부에 불필요하게 노출되면 의도치 않은 호출이 생기고 리팩토링 범위가 넓어집니다. Codex만 이를 지켰습니다.

Claude ❌ 미적용
내부 Composable에 private 없음. CitySelector, WeatherCard 등이 파일 외부로 노출됨. 
@Composable
fun CitySelector(...)  // public — 의도치 않은 노출
@Composable
fun WeatherCard(...)   // public
Codex ✅✅ 완벽
파일 내부 Composable 전부 private. 외부에서 쓰는 컴포넌트(ForecastRow, MetricTile)만 별도 파일로 공개. 
@Composable
private fun WeatherHeader(...)   // ✅
@Composable
private fun HourlySection(...)   // ✅
Gemini ❌ 미적용
내부 Composable에 private 없음. 게다가 와일드카드 import 사용으로 어떤 심볼이 실제로 쓰이는지 파악이 어려움. 
import androidx.compose.material3.*
import androidx.compose.foundation.layout.*
// 어떤 심볼이 쓰이는지 불명확
1-4 UiState에 계산 프로퍼티 넣기

UI에서 매번 계산하는 값이 있다면 data class UiState 안에 계산 프로퍼티로 정의하면 중복이 사라집니다. Codex만 이 패턴을 사용했습니다.

❌ Claude / Gemini — UI 레이어에서 매번 계산
// WeatherScreen.kt 여러 Composable에서 반복
val weather =
    uiState.weatherList[uiState.selectedIndex]
val isFavorite =
    uiState.selectedIndex == uiState.favoriteIdx
// 같은 코드가 여러 곳에 중복됨
✅ Codex — UiState에 computed property 정의
data class WeatherUiState(
    val selectedCityIndex: Int = 0,
    val weatherList: List<WeatherData> = ...,
    val favoriteCityIndex: Int? = null
) {
    val selectedWeather
        get() = weatherList[selectedCityIndex]
    val isSelectedCityFavorite
        get() = selectedCityIndex == favoriteCityIndex
}
1-5 렌더링 성능 패턴

Column + forEach는 모든 아이템을 한 번에 그립니다. 아이템이 늘어날수록 LazyColumn이 필수입니다. Gemini만 일관되게 Lazy를 사용했습니다.

항목 Claude Codex Gemini

시간별 예보 목록 LazyRow ✅ horizontalScroll + Row ❌ LazyRow ✅
상태바 패딩 enableEdgeToEdge() ✅ statusBarsPadding() ✅ padding(top=48.dp) ❌ 하드코딩

1-6 패키지 구조

MVVM 프로젝트는 data/, viewmodel/, ui/ 레이어별 폴더 분리가 기본입니다. 세 AI가 서로 다른 구조를 선택했습니다.

레이어 분리 구조 자체는 가장 올바름. 단 WeatherScreen.kt가 340줄 단일 파일이고, conditionEmoji() 같은 유틸 함수가 UI 파일 최상단에 노출됨.

03

바이브 코딩 체크리스트 — Before / During / After

AI 코드 생성 전후로 팀이 지켜야 할 규칙을 3단계로 정리했습니다. 노란색 항목이 실험에서 가장 많이 놓친 항목입니다.

BEFORE 프롬프트 작성 전 — 팀 규칙 먼저 설정
아키텍처 레이어를 명시한다. "data/, viewmodel/, ui/ 폴더 구조를 따를 것"을 프롬프트에 포함. 지시 없으면 루트에 모든 파일을 넣는다.
상태 관리 방식을 지정한다. "StateFlow + _uiState.update{} 패턴 사용" 명시. AI마다 기본값이 다르고 안전성도 다르다.
Composable 설계 원칙을 지정한다. "ViewModel 직접 주입 금지 — UiState + 콜백 파라미터 방식 사용", "내부 Composable은 private 선언"
금지 패턴을 명시한다. "collectAsStateWithLifecycle() 사용", "와일드카드 import 금지", "하드코딩 dp 패딩 금지"
파일 크기 제한을 안내한다. "한 파일 200줄 초과 시 components/ 로 분리" — 지시 없으면 모든 UI를 한 파일에 넣는다.
Theme 파일 위치를 지정한다. "ColorScheme는 ui/theme/Theme.kt에 정의할 것" — 미지정 시 화면 파일 안에 섞인다.
DURING 생성 중 — 실시간 확인
파일 생성 경로를 확인한다. 파일이 어느 폴더에 생성되는지 실시간 확인. 루트에 몰리면 즉시 수정 요청.
ViewModel 상태 패턴을 확인한다. _uiState.value = 패턴이 보이면 .update {}로 수정 요청.
WeatherScreen 시그니처를 확인한다. fun WeatherScreen(viewModel: ViewModel)이면 fun WeatherScreen(uiState, callbacks)로 수정 요청.
import 목록을 확인한다. import androidx.compose.material3.* 형태의 와일드카드가 나타나면 명시적 import로 교체 요청.
AFTER 커밋 전 — 리뷰 필수 항목
collectAsStateWithLifecycle() 여부 확인. collectAsState()가 있으면 전부 교체. Gemini 생성 코드에서 자주 발생.
하드코딩 값 검색. padding(top = **.dp), 도시 목록 리터럴이 Composable 내부에 있으면 제거.
private 수식자 확인. 외부에서 사용하지 않는 Composable에 private 추가. Claude, Gemini 생성 코드에서 빠져 있음.
유틸 함수 위치 확인. conditionEmoji() 같은 순수 변환 함수가 UI 파일에 있으면 data 레이어로 이동.
경계값 검사 추가. 인덱스로 리스트 접근하는 함수에 if (index in list.indices) 추가.

이번 실험의 핵심 교훈: 세 AI 모두 규칙을 주지 않으면 제각각으로 만든다. CLAUDE.md 같은 규칙 파일로 팀 컨벤션을 한 번만 작성해두면 매번 프롬프트에 반복하지 않아도 됩니다.

CLAUDE.md / .cursorrules 활용법
프로젝트 루트에 AI 지시 파일을 작성해두면 Claude Code, Cursor 등 대부분의 AI 도구가 해당 규칙을 자동으로 참고합니다. 이 파일을 git에 커밋해 팀 전체가 공유하세요. 
## Kotlin / Jetpack Compose 컨벤션 (CLAUDE.md 예시)

### 아키텍처
- 패키지 구조: data/ · viewmodel/ · ui/ · ui/theme/ · ui/components/
- ViewModel은 StateFlow + _uiState.update{} 패턴만 사용
- Composable에 ViewModel 직접 주입 금지 → UiState + callback 파라미터 방식 사용

### Composable 규칙
- 파일 외부에서 사용하지 않는 Composable은 private 선언 필수
- 한 파일 200줄 초과 시 components/ 에 파일로 분리
- 와일드카드 import 금지 (import xxx.* 사용 금지)

### 안전성
- collectAsStateWithLifecycle() 사용 (collectAsState() 금지)
- 인덱스 기반 리스트 접근 전 if (index in list.indices) 검사 필수
- 하드코딩 dp 패딩 금지 → statusBarsPadding() / WindowInsets 사용

### 금지 패턴
- _uiState.value = _uiState.value.copy() → .update {} 로 교체
- Composable 내부에 데이터 리터럴 하드코딩 금지
- ColorScheme 정의를 화면 파일에 넣지 말 것 → ui/theme/Theme.kt 에만

코드 리뷰 체크리스트 — 커밋 전 11개 항목

이번 실험에서 실제로 발견된 문제를 기반으로 작성했습니다. 발견 위치를 함께 기록해 어떤 AI가 어디서 놓쳤는지 확인할 수 있습니다.

1
StateFlow 업데이트는 .update {} 사용
_uiState.value = _uiState.value.copy(...)는 비동기 환경에서 race condition을 일으킬 수 있습니다. _uiState.update { it.copy(...) }로 교체하세요.
발견: Claude WeatherViewModel.kt — selectCity(), toggleFavorite(), refresh() 전체
2
collectAsStateWithLifecycle() 사용 확인
collectAsState()는 앱이 백그라운드에 있을 때도 Flow를 수집해 불필요한 리소스 낭비와 리컴포지션을 일으킵니다.
발견: Gemini WeatherScreen.kt — val uiState by viewModel.uiState.collectAsState()
3
Composable 파라미터에 ViewModel 직접 주입 금지
ViewModel을 파라미터로 받으면 @Preview 사용이 불가능하고 단위 테스트가 어렵습니다. UiState + 콜백 파라미터 방식을 사용하세요.
발견: Claude / Gemini WeatherScreen.kt — fun WeatherScreen(viewModel: WeatherViewModel)
4
내부 Composable에 private 선언
파일 외부에서 사용하지 않는 Composable은 private을 붙이세요. 노출 범위를 좁혀 리팩토링이 쉬워집니다.
발견: Claude / Gemini WeatherScreen.kt — CitySelector, WeatherCard, ForecastItem 등
5
Composable 내부 데이터 하드코딩 금지
도시 목록 등 데이터가 Composable 함수 내부에 리터럴로 있으면 UiState로 이동해야 합니다. Composable은 그리는 역할만 해야 합니다.
발견: Gemini WeatherScreen.kt — val cities = listOf("서울", "부산", "제주")
6
statusBarsPadding() / WindowInsets 사용
상태바 높이를 dp로 하드코딩하면 기기마다 레이아웃이 깨집니다. statusBarsPadding() 또는 WindowInsets.statusBars를 사용하세요.
발견: Gemini WeatherScreen.kt — .padding(top = 48.dp)
7
Theme / ColorScheme 위치 확인
darkColorScheme(), MaterialTheme() 정의는 ui/theme/Theme.kt에만 있어야 합니다. 화면 파일 안에 넣으면 다른 화면에서 재사용이 불가능합니다.
발견: Codex WeatherScreen.kt — private val AppColorScheme = darkColorScheme(...)
8
유틸 함수의 레이어 위치 확인
날씨 조건 → 이모지, 조건 → 색상 같은 순수 변환 함수는 UI 파일이 아닌 data 레이어에 있어야 합니다.
발견: Claude WeatherScreen.kt — top-level fun conditionEmoji(), conditionGradient()
9
인덱스 접근 전 경계값 검사
리스트를 인덱스로 접근하는 함수에는 if (index in list.indices) 검사를 추가하세요. AI는 종종 이 검사를 생략합니다.
모범: Codex selectCity() — if (index in uiState.weatherList.indices)
10
와일드카드 import 제거
import androidx.compose.material3.* 형태는 어떤 심볼이 실제로 쓰이는지 알 수 없습니다. 명시적 import로 교체하세요.
발견: Gemini WeatherScreen.kt — import androidx.compose.foundation.layout.* 등 다수
11
LazyColumn / LazyRow 사용 여부
아이템 수가 고정되어 있어도 Column + forEach 대신 LazyColumn을 사용하는 것이 안전합니다. 나중에 서버 데이터로 교체할 때 성능 문제가 없습니다.
발견: Claude / Codex — 5일 예보를 Column + forEach로 렌더링
06

결론

AI가 만든 코드를 그대로 쓰면 안 된다. 하지만 규칙을 주면 달라진다.

세 AI 모두 동일한 스펙을 받았지만 코드 품질이 각각 달랐습니다. 어떤 AI도 처음부터 모든 규칙을 완벽히 따르지 않았습니다. 반대로 말하면 — 규칙을 명시하면 어느 AI도 팀 컨벤션에 맞는 코드를 만들어낼 수 있습니다. CLAUDE.md 같은 규칙 파일 하나가 매 PR 리뷰 시간을 줄여줍니다.
